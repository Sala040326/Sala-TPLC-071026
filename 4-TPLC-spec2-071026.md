# [2]醫療器材全生命週期TPLC管理系統規格書

## 1-1 前言

在生醫資通訊（Bio-ICT）與跨國監管科技（RegTech）高度交織的 2026 年，醫療器材產業面臨前所未有的全球合規挑戰。全球衛生主管機關如美國食品藥物管理局（FDA）、歐盟醫療器材協調小組（MDCG）、台灣衛生福利部食品藥物管理署（TFDA）等，對於上市後監管（Post-Market Surveillance, PMS）與全生命週期（Total Product Life Cycle, TPLC）的管理力道正呈指數級增長。歐盟醫療器材法規（MDR 2017/745）的全面強制實施，以及美國 FDA 對於醫療器材網路安全（Cybersecurity）與人工智慧醫療軟體（SaMD）的最新指引，皆要求製造商建立動態、即時且可追溯的風險管理體系。

傳統的醫療器材上市後管理依賴大量的人工審查、紙本比對與離線資料庫查詢，導致法規情報落後、產品不良事件（Adverse Events）通報延遲、唯一器械識別（UDI）報送錯誤率高。為了突破此產業痛點，本系統旨在建構一套**智慧化醫療器材全生命週期TPLC管理系統**。本系統深度整合全球醫療法規自動化生成與變更洞察系統（GRAS-Engine v2.4.0-Prod），應用先進的前沿人工智慧（AI）技術強化醫療器材上市後管理。

本系統的設計與架構深度參考了國際前沿衛生主管機關的 AI 應用案例與監管框架：

* **美國 FDA HALO/ELSA 4.0**：參考其利用大型語言模型與自然語言處理技術，自動化掃描、分類與評估醫療器材不良事件通報（MAUDE 資料庫）之機制，建立高度敏感的風險預警指標。
* **瑞士 Swissmedic**：借鑑其在多語系法規環境下，利用 AI 進行跨國安全警訊（Safety Alerts）與醫療器材回收資訊（Field Safety Notice, FSN）的自動化對齊與語意分析架構。
* **新加坡 HSA**：參考其動態風險分級評估與主動式上市後監督抽驗模型，優化系統在面對變更申請時的審查漏洞補強演算法。

本系統不僅是一套資料庫管理工具，更是一個企業級的「合規大腦與雙向協同管理平台」，專為 Class II/III 高風險有源醫療器材及 SaMD（醫療器材軟體，如智慧心電貼片監控系統）打造。透過多主體 AI 代理人技術（Multi-Agent Collective Intelligence），實現從全球不良事件情報監測、臨床風險報告審閱、智慧 UDI 標籤動態序列化修復、到跨國衛生當局 XML/JSON 申報 Payload 自動編譯驗證的完整閉環，確保醫療器材在上市後生命週期中的絕對合規與安全。

## 1-2 現況（盤點現有資訊/資料集）

為達成全生命週期的動態監管，系統必須整合與清洗來自內外部、結構化與非結構化的多源異質資料庫。以下為本系統核心盤點並納入管轄的資訊與資料集架構：

```
                              [TPLC 資料匯流中樞]
                                      |
         +----------------------------+----------------------------+
         |                            |                            |
  [SQL 結構化資料庫]          [NoSQL 非結構化文獻]        [外部跨國衛生當局資源]
  - 醫療器材許可證資料         - 醫療器材安全警訊          - FDA Recall / MAUDE
  - UDI/GTIN-14 核心屬性      - 醫療器材回收/FSN文書       - Swissmedic ADR / Alerts
  - 分類分級/品項代碼          - 臨床文獻與客訴紀錄          - TGA / Health Canada 警訊
  - 來源流向申報資料

```

### 1. SQL 結構化資料

結構化資料庫主要儲存法規邊界明確、具備固定欄位格式的核心主數據，採用高可用性關聯式資料庫進行動態查詢與關聯：

* **醫療器材許可證資料庫**：包含許可證字號、中英文品名、製造廠名稱/地址、申請商資訊、核發日期、有效日期、品項代碼及核定之預期用途等核心欄位。
* **醫療器材單一識別系統（UDI）資料庫**：包含全球貿易品項識別碼（GTIN-14）、器械識別碼（DI）、包裝層級（Packaging Level）、是否有批號（Lot）/序列號（Serial）、軟體版本號、以及發行機構（GS1 / HIBCC）等標準化元數據。
* **醫療器材分類分級資料庫**：收錄美、歐、台三地之品項分類對照表（如 FDA Procode、EU CND Code、TFDA 分類分級品項代碼），定義各品項之基礎風險等級（Class I, IIa, IIb, III）。
* **醫療器材來源流向申報資料**：整合企業內部 ERP/MES 系統與官方申報介面，追蹤特定批號/序列號之醫材出廠日期、進口報單、醫療機構銷售流向及庫存狀態。

### 2. NoSQL 非結構化資料

上市後監管的大量核心線索散落於文字、報告與合規文獻中，系統採用分散式 NoSQL 文件資料庫儲存，以利進行大語言模型（LLM）的向量化與語意檢索：

* **醫療器材安全警訊（Safety Alerts）**：國內外主動發布之安全性通報、不良反應統計分析報告、以及原廠對醫事人員之溝通說明信（DHCP Letter）。
* **醫療器材回收資訊（Field Safety Corrective Action, FSCA / Recall）**：包含醫療器材回收原因說明書、受影響之批號清單、矯正預防措施（CAPA）報告書、以及向主管機關提交之回收完成報告。
* **其他內部非結構化文本**：包含臨床評價報告（CER）、上市後臨床跟隨（PMCF）報告、使用者客訴紀錄（Customer Complaints）以及維修保養日誌。

### 3. 其他外部資源（非食品藥物管理署資料 / 網路實時資源）

系統透過定時網路爬蟲與衛生當局開放 API（如美國 FDA openFDA 介面），實時導入全球監管動態，並透過智慧情報代理人進行對齊：

* **FDA Recall Database & MAUDE**：美國 FDA 的醫療器材召回資料庫與製造商和使用者安全性器械經驗資料庫（MAUDE），擷取全球有源醫材的不良事件回報。
* **Swissmedic ADR / Safety Alerts**：瑞士醫療器材主管機關發布之藥物與醫材不良反應（ADR）報告與最新安全聲明。
* **TGA Safety Alerts**：澳洲醫療網路與治療商品管理局（TGA）發布之危害通報與合規指引變更。
* **Health Canada Field Correction Notice**：加拿大衛生部發布之現場矯正通知與產品下架數據。

## 1-3 使用情境（應用於醫療器材上市後管理）

本系統針對智慧心電貼片監控系統（SaMD / Class IIa Wearable Device）及各類高風險醫材，設計了 6 個涵蓋全生命週期、著重於上市後管理的綜合使用情境（Use Cases）。以下詳述其操作流程與 AI 協同機制：

### Use Case 1: 全球不良事件（Adverse Event）即時語意審查與自動化風險分級

* **情境描述**：當系統從美國 FDA MAUDE 或瑞士 Swissmedic 自動擷取到一則關於「穿戴式心電訊號於高濕度環境下發生斷連，導致致命性心律不整漏報」的不良事件通報時，情報代理人立即啟動。
* **操作流程**：
1. 系統自動對非結構化葡文/英文通報進行語意感知分塊（Semantic Chunking）與醫療法學專用語庫（MedReg-Glossary）對齊。
2. AI 審查官（Auditor Sentiment Simulator）掃描該事件描述，比對企業內部智慧心電貼片的現有技術檔案。
3. 系統自動計算出該事件與本廠產品的技術關聯度，評估是否屬於「嚴重不良事件（SAE）」。
4. 自動將風險評估結果填入上市後監管（PMS）儀表板，並在判定達到危害臨界點時，自動生成跨國衛生當局通報草稿。



### Use Case 2: 跨國法規變更（如 ANVISA RDC 80/2016）設計適用性雙向衝擊分析

* **情境描述**：巴西主管機關 ANVISA 發布了關於醫療器材 UDI 登載與標籤規範的修訂版技術標準（RDC 80/2016），要求標籤必須加註本地持證代表（BRH）資訊。
* **操作流程**：
1. 監管情報 RAG 代理人偵測到法規變更，提取出 RDC 80/2016 的核心條款。
2. 系統調用當前智慧心電貼片的技術上下文中「在巴西銷售」且「風險等級為 Class II」的屬性。
3. AI 發起**設計適用性分析（Design Alignment Synthesis）**，發出高亮警示：「當前包裝標籤缺少巴西持證代表登記編號與法律聯絡地址」。
4. 用戶點選一鍵修復，系統將對應的補強條款（Remediation Clause）雙向同步至企業內部的合規報告中。



### Use Case 3: 現場矯正措施（FSCA）受影響批號追溯與來源流向自動化申報

* **情境描述**：生產線反饋某特定批號（例如 LOT999）的心電貼片導電膠存在黏性失效風險，企業必須主動發起現場安全矯正措施（FSCA）。
* **操作流程**：
1. 品質經理在系統內輸入受影響批號 `LOT999`。
2. 系統自動調取 **SQL 結構化資料庫中的來源流向申報資料**，跨表查詢出該批次貼片的總生產量、已出庫數量、以及銷往各大醫療機構的具體明細。
3. 申報編譯代理人自動將銷流數據封裝，編譯為符合各國衛生主管機關格式的回收通報 Payload。
4. 系統自動產出客戶通知信（Customer Letter）與產品回收進度追蹤表。



### Use Case 4: 智慧 UDI 標籤向量渲染與動態序列化（PI）靜態規格校驗

* **情境描述**：工廠即將上線生產新一批智慧心電貼片，需要動態生成包含製造日期、失效日期、批號與唯一序列號的 UDI 二維條碼標籤。
* **操作流程**：
1. 前端控制層（Vite+React19）調用**智慧 UDI 貼紙設計引擎（AIDC Vector Engine）**。
2. 系統自動抓取 DI（GTIN-14：`00850001234567`），並動態注入生產識別碼（PI）：製造日期（11）、失效日期（17）、批號（10：`LOT999`）、序列號（21：`SN8877`）。
3. 靜態校驗器（Static Validator）執行語法樹分析（AST Parsing），驗證 DI 長度是否精確為 14 位，且失效日期是否大於製造日期。
4. 通過驗證後，直接以 SVG 高保真度向量路徑（`<rect>` 與 Finder Pattern）在前端渲染 GS1 DataMatrix，提供給工廠 Zebra 600dpi 噴印設備，杜絕因條碼長度或格式錯誤導致的退件。



### Use Case 5: 軟體演算法變更（SaMD 升級）引發之 UDI-DI 重新申報臨界點預測

* **情境描述**：研發團隊為智慧心電貼片引進了全新 AI 深度學習心律不整偵測演算法（涉及 SaMD 重大韌體變更），需要評估是否需向美國 FDA 重新申請 510(k) 或更新 UDI-DI。
* **操作流程**：
1. 研發工程師在協同編輯器中輸入演算法結構變更參數。
2. 合規編排代理人（Orchestrator）接收事件，調用歷史 510(k) 與 MDR 退件資料庫進行混合檢索。
3. 系統評估該變更屬於「影響產品預期用途與核心診斷演算法之重大變更」，並非單純的軟體修補（Bug Fix）。
4. 預測性核准率儀表板（Predictive FTR Engine）顯示：若不重新申報 UDI-DI 而直接強制上架，致命退件機率高達 94.5%。系統自動引導用戶進入全新 UDI-DI 的編譯與申報工作流。



### Use Case 6: 上市後監管（PMS）技術文件一鍵式 AI 自動修補與高階 PDF 導出

* **情境描述**：企業準備向歐盟公告機構（Notified Body）提交年度定期安全性更新報告（PSUR），但被系統偵測到缺少對最新臨床演算法數據庫多樣性（偏見驗證）的文字宣告。
* **操作流程**：
1. 系統自動標註出該項 **MAJOR 級別合規漏洞**。
2. 用戶點選「一鍵 AI 補強」，審查漏洞補強代理人（Audit Agent）依據符合 IEC 62304 規範的核心邏輯，自動生成中英對照的修正條款（Corrective Clauses）。
3. 協同 Rich Text 編輯器自動將文字無縫無感地追加（Append）至報告書底層。
4. 串接後端 Puppeteer 引擎，系統應用 `@media print` 媒介標準與分頁防斷裂（Page Break）機制，將包含高保真度 UDI 向量貼紙與合規免責浮水印的技術文件，導出為符合 A4 標準規格的工業級 PDF 報告。

## 1-4 系統架構 Architecture

本系統採用現代化、高效能的前後端分離（Decoupled Full-Stack）與事件驅動架構（Event-Driven Architecture, EDA）。前端基於 Vite 與 React 19 構建極致流暢的使用者介面，後端基於 Node.js 與 Express 提供企業級的法規與資料編譯服務。核心法規合規與數據處理透過「多主體 AI 代理人網路」進行縱深編排。

### 1. 系統資料與代理人協同編排流（System Data & Agent Orchestration Flow）

```
[前端控制層 App.tsx (Vite+React19)] <--- WebSocket / REST-API ---> [後端服務層 server.ts (Express)]
        |                                                                |
        +---> (1) 智慧 UDI 貼紙設計引擎 (AIDC Vector Engine)                   +---> (A) Gemini 2.5 Flash / Pro 
        +---> (2) 協同式 Rich Text 編輯器與合規審閱模組                              |   (網頁即時 RAG 檢索增強)
        +---> (3) 預測性核准率儀表板 (Predictive FTR Engine)                    +---> (B) 歷史 510(k) 與 MDR 退件資料庫
        +---> (4) GUDID/EUDAMED XML/JSON 編譯校驗器                             +---> (C) PDF 印刷排版與 Puppeteer 引擎

```

### 2. 多主體 AI 代理人技術架構（Multi-Agent Collective Intelligence）

系統將複雜的法規合規任務拆解至多個高度專業化、具備工具調用（Tool Calling）與長期記憶能力的子代理人中：

```
                         +------------------------------------+
                         |    合規編排代理人 (Orchestrator)    |
                         +------------------------------------+
                                           |
              +----------------------------+----------------------------+
              |                            |                            |
     +--------v--------+          +--------v--------+          +--------v--------+
     | 監管情報 RAG 代理 |          | 審查漏洞補強代理 |          | 申報編譯驗證代理 |
     | (Intel Agent)   |          | (Audit Agent)   |          | (Payload Agent) |
     +-----------------+          +-----------------+          +-----------------+

```

* **合規編排代理人（Orchestrator Agent）**：負責解析用戶的初始設計規格、客訴事件或變更目標，並將任務合理路由派發至子代理人。採用 Plan-and-Solve 機制。
* **監管情報 RAG 代理人（Regulatory Intelligence Agent）**：負責連線全球衛生當局資料庫進行混合檢索，提取最新的法規宣告與安全指引，提供精確的法學翻譯與適用性比對。
* **審查漏洞補強代理人（Audit & Mitigation Agent）**：扮演衛生當局審查官（Auditor Sentiment Simulator），利用深度學習預測模型分析技術文件結構，標註致命缺失並自動生成修正條款。
* **申報編譯驗證代理人（Schema Compiler & Validator Agent）**：負責提取 UDI 標籤與 PMS 數據中的動態屬性，將其結構化編譯為 HL7 SPL XML 或 EUDAMED JSON Payload，並進行靜態語法樹安全掃描。

## 1-5 使用需求 SRS

為了確保系統的功能完整性，軟體需求規格（Software Requirements Specification, SRS）劃分為四大核心功能模組：

### 1. 醫療器材全生命週期資料整合模組（TPLC Data Hub）

* **SRS-001：多源資料同步**。系統必須提供排程（Cron Job）及即時 Webhook 機制，定時同步官方醫療器材許可證、UDI 資料庫及外部 FDA MAUDE、Swissmedic 不良事件數據。
* **SRS-002：異質資料清洗與對齊**。系統必須自動將非結構化的安全警訊進行語意分塊（Semantic Chunking），並強植多維度元數據（Metadata Tagging），包含法規管轄區（Jurisdiction）、適用風險等級、執行日期與文件狀態。
* **SRS-003：合規詞彙映射**。系統在處理跨國法規文獻時，必須調用內建之 `MedReg-Glossary` 專用語庫，確保葡文、日文等外文條款能精確對齊繁體中文的醫學法學專有名詞。

### 2. 智慧 UDI 標籤生成與動態序列化引擎（AIDC Labeling Engine）

* **SRS-004：GS1 應用識別碼（AI）標準對齊**。系統必須嚴格遵循國際 UDI 格式對動態數據（Production Identifier, PI）進行序列化封裝，完整支援 GS1-128 與 GS1 DataMatrix 標準。
* **SRS-005：高保真度向量渲染**。前端必須使用 React 配合動態 SVG 路徑技術繪製二維條碼。條碼必須採用 14x14 至 26x26 模組矩陣，利用座標偏移量像素級繪製 `<rect>` 方塊與 L 型對齊邊界（Finder Pattern），確保下載的 SVG 圖檔在工業級 600dpi 打印下不失真。
* **SRS-006：動態變量封裝**。系統必須即時解析器械識別碼（DI/GTIN-14）、製造日期（AI: 11）、失效日期（AI: 17）、批號（AI: 10）與序列號（AI: 21），並隨前端表單修改進行毫秒級實時渲染更新。

### 3. 預測性首發核准率與審查漏洞修補模組（Predictive FTR & Audit Engine）

* **SRS-007：審查風險機率預測**。系統必須提供一個模擬卷積神經網路（CNN）特徵擬合結果的預測模型。模型需掃描當前上市後監管報告的結構完整度，評估出基礎首發核准概率（First-Time-Right %）。
* **SRS-008：一鍵式 AI 自動修復補正**。當系統標註出重大合規漏洞（如缺少 Basic UDI-DI、臨床演算法數據庫多樣性不足等）時，必須提供「一鍵 AI 補強」功能。點選後，系統需透過狀態提升（State Lifting）調升核准機率，並在背景將 Remediation Clause 自動且無縫地追加（Append）至中央協同編輯器中。
* **SRS-009：雙向狀態同步與動效回饋**。代理人之間必須透過事件驅動架構（EDA）的中央狀態總線（State Bus）進行通訊。當漏洞修復完成時，除更新文檔外，前端需同步觸發 Confetti 七彩紙屑等高滿意度動效。

### 4. 衛生當局申報 Payload 編譯與校驗模組（Submission & Validator Engine）

* **SRS-010：FDA GUDID (HL7 SPL XML v3) 編譯**。系統必須能將標籤元數據自動編譯為符合美國 FDA 規定的 HL7 Structured Product Labeling (SPL) v3 XML 格式，核心節點必須包含 `<primaryDI>`、`<deviceCharacteristics>` 及 `<productionIdentifiers>`。
* **SRS-011：EUDAMED JSON Payload 生成**。系統必須能將數據編譯為符合歐盟 EUDAMED Device Submission 規範的 JSON Schema，核心節點需包含 `basicUdiDi` 映射、製造商歐洲單一登記號（SRN，系統預設為 `EU-SRN-TW2026999`）及符合 MDR Annex VI 的動態 PI 欄位。
* **SRS-012：資料完整性靜態校驗**。系統必須內建靜態校驗器（Validator），在 Payload 送出前進行抽象語法樹（AST）安全掃描，強制驗證 GTIN-14 位數長度、批號/序列號佔位符同步、以及失效日期是否大於製造日期。若失敗，需回傳詳盡的缺陷錯誤代碼。

## 1-6 特殊需求

本章節定義全球醫療法規自動化生成與變更洞察系統（GRAS-Engine v2.4.0-Prod）在企業級技術架構、產品規格、安全性及數據隱私方面的特殊設計規格：

### 1. 混合檢索架構（Hybrid Retrieval & Re-ranking）

為了徹底杜絕大語言模型在面對極具深度與時效性的醫療法規時可能發生的「幻覺（Hallucination）」問題，系統採用了高精密度的 **Sparse-Dense Hybrid Search** 與重排機制：

* **密向量檢索（Dense Retrieval）**：使用 `Google text-multilingual-embedding-002` 模型對全球法規條文與安全警訊進行向量化，捕捉條款背後的監管意圖（例如「標籤可讀性」與「AIDC 條碼對比度」的內在語意關聯）。
* **稀疏向量檢索（Sparse Retrieval）**：基於 `BM25` 演算法，對特定法規編號（如 "21 CFR 801.20"）、專有名詞與 UDI 應用識別碼（如 "01", "17"）進行精確匹配。
* **重排器（Re-ranker）**：混合檢索後，利用 Cross-Encoder 交叉注意力模型進行結果重排，篩選出與當前智慧醫材設計相關度最高的前 5 條法規片段，強行注入 LLM 的 System Instructions 中。

### 2. 數據去識別化（De-identification）與安全邊界保護

在跨國監管環境中，患者隱私與企業智慧財產權是不可妥協的紅線：

* **患者隱私（HIPAA & GDPR 合規）**：系統在處理上市後客訴、維修紀錄或臨床報告審閱時，內建敏感數據自動遮蔽引擎（DLP）。任何涉及患者姓名、病歷號、身份證號或具體心電波形之元數據，在進入 AI 代理人或上傳至服務器前，會自動被去識別化並進行 SHA-256 單向哈希加密。
* **企業機密保護**：智慧心電貼片的底層專利硬體電路圖、私有診斷演算法原始碼均被限制於本地 React 沙箱中處理。上傳至後端 `/api/generate` 代理路由的，僅限於 UDI 元數據與非敏感法規條款，徹底將企業核心核心技術與公有雲 AI 隔離。
* **決策免責宣告與 AI 輔導邊界（AI Liability Safeguard）**：本系統定位為「合規決策輔助工具（DSS）」，而非法律實體裁決者。所有生成的 XML/JSON 申報 Payload 及 RAG 對照報告末尾均會自動加蓋帶有系統時間戳（Timestamp 2026-07-10）與 AI 特徵指紋的免責浮水印，明確提示製造商最終合規審核權仍歸屬於企業內部授權簽字人（PRRC / Quality Manager）。

### 3. 前端工程、企業整合與高階列印規範

* **前後端介面架構**：後端基於 Node.js 與 Express 構建，透過 `@google/genai` 官方最新 SDK 呼叫 `gemini-2.5-flash`（或用戶自選的 `gemini-2.5-pro` 模型），並啟用 `Google Search Grounding` 行動檢索。前端與後端採用 WebSocket 與 REST-API 進行雙向非同步通訊。
* **企業 PDF 導出與 Puppeteer 高階列印設置**：系統整合 Puppeteer 無頭瀏覽器。當觸發導出時，後端會將經過 HTML Tailwind 格式美化後的 Rich Text 內容，以 `@media print` 媒介標準渲染為 A4 標準規格 PDF 文件。系統設置面板中提供「企業官方合規浮水印（Watermark）」開關、自定義頁首頁尾配置、以及文字大小微調，確保與傳統印刷格式無縫接軌。
